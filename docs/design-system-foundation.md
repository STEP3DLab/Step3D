# Единая дизайн-система: консистентность и доступность

## 1) Цели

- обеспечить визуальную и поведенческую консистентность интерфейса;
- сократить time-to-market новых фич;
- уменьшить количество UI/UX-дефектов;
- встроить доступность (a11y) как стандарт разработки.

---

## 2) Состав дизайн-системы

## A. UI-kit (Figma)

Минимальный состав:
- цветовые стили (семантические, брендовые, сервисные);
- типографическая шкала;
- spacing/grid/elevation;
- иконографика;
- компоненты и их варианты;
- шаблоны ключевых экранов.

Правила:
- только семантические стили (без «hardcoded» значений в макетах);
- обязательные variant properties для состояний;
- автолэйаут по умолчанию.

## B. Design Tokens

Слои токенов:
1. **Primitive** (raw values: color, size, radius, font-size);
2. **Semantic** (bg/surface/text/border/success/error/warning);
3. **Component tokens** (button-primary-bg, input-border-focus и т.д.).

Обязательные группы токенов:
- color;
- typography;
- spacing;
- radius;
- shadow/elevation;
- motion (duration/easing);
- z-index.

Формат поставки:
- единый source-of-truth (JSON + Style Dictionary/аналог);
- синхронизация в Figma и кодовые артефакты (CSS variables / platform tokens).

## C. Библиотека компонентов

Приоритет v1:
- Button;
- Input/Textarea;
- Select/Dropdown;
- Checkbox/Radio/Switch;
- Modal/Drawer;
- Tooltip;
- Tabs;
- Toast/Alert;
- Table/List;
- Pagination;
- Breadcrumbs;
- Empty/Error/Loading states.

Для каждого компонента обязательны:
- API (props, slots, events);
- состояния (default/hover/focus/active/disabled/loading/error/success);
- a11y-спека (roles, aria, keyboard navigation, focus management);
- do/don’t примеры;
- критерии готовности к продакшену.

## D. Гайдлайны по состояниям и ошибкам

Единые правила:
- ошибка объясняет причину + действие по исправлению;
- inline validation + summary для длинных форм;
- пустые состояния дают следующий шаг;
- loading всегда с ожидаемым временем/сигналом прогресса;
- системные ошибки: человекопонятный текст + recovery action.

---

## 3) Доступность (A11y by default)

Базовый стандарт:
- WCAG 2.2 AA для ключевых пользовательских путей.

Контрольный чеклист:
- контраст текста и интерактивных элементов;
- видимый focus state;
- управление с клавиатуры без мыши;
- корректная семантика и aria-атрибуты;
- логичный tab order;
- доступные имена для controls;
- ошибки и подсказки читаются скринридерами.

Процесс:
- автоматические проверки в CI (lint/a11y checks);
- ручной аудит перед релизом критических экранов;
- обязательный accessibility review в дизайн-ревью.

---

## 4) Модель управления (DesignOps)

Роли:
- **Design System Owner** (приоритеты и roadmap);
- **UX Lead** (качество взаимодействий);
- **Frontend Lead** (техническая интеграция);
- **QA/A11y Champion** (валидность и доступность).

Процесс изменений:
1. RFC на новый компонент/изменение;
2. дизайн-ревью + техревью;
3. реализация + документация;
4. релиз версии;
5. миграция потребителей.

Версионирование:
- SemVer для компонентов и токенов;
- changelog + migration guide;
- deprecation policy (срок и путь перехода).

---

## 5) KPI дизайн-системы

- снижение UI-багов на релиз;
- доля экранов, собранных из DS-компонентов;
- среднее время доставки новой UI-фичи;
- снижение количества кастомных «одноразовых» компонентов;
- рост внутренних UX-оценок (design/peer review score).

---

## 6) Дорожная карта запуска (10 недель)

- **Недели 1–2:** аудит текущего UI, инвентаризация паттернов и расхождений.
- **Недели 3–4:** архитектура токенов, базовые стили, theme strategy.
- **Недели 5–6:** компонентная библиотека v1 (core components).
- **Недели 7–8:** документация, примеры, a11y-гайдлайны.
- **Недели 9–10:** миграция приоритетных экранов + метрики adoption.
